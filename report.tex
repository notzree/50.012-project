\begin{abstract}
As Large Language Models (LLMs) transition from research to production, developers face a critical problem: the lack of standardized tooling to evaluate the real-world network and system performance of LLM APIs. Existing benchmarks focus primarily on model reasoning capabilities rather than operational metrics like tokens per second (TPS), latency, and Time to First Token (TTFT). This report details our design and development of an "AI Speed Test", a Next.js-based web application capable of empirically benchmarking major LLM providers (e.g., OpenAI, Anthropic, Google) to inform production deployment decisions. Our approach bridges key networking concepts such as Application Layer behavior, Transport Layer properties, and Multimedia Streaming paradigms to standardize AI API evaluation.
\end{abstract}

\section{Introduction \& Problem Statement}
\subsection{Overall Problem Domain, Background, and Motivation}

The emergence of commercial Large Language Models (LLMS) has introduced a new paradigm to the software engineering world, where the primary method of applying these LLMs is via third-party APIs. When using these APIs, it is apparent that developers lack a standardized framework that can be used to evaluate the performance and usability of these models from a networks and systems perspective. 

Performance benchmarks are slowly emerging and reinforcing itself as a strategic infrastructure. This can be outlined during a recent exchange in which Accenture acquired Ookla, the company behind Speedtest, Downdetector, Rootmetrics, and Ekahau, for more \$1 billion. The further AI applications integrate itself as a standardized tool, it will transition into being real-time and client-facing, emphasizing the importance of latency and throughput in terms of user experience. Organizations will urgently require a ``Speedtest-like`` solution in order to benchmark and regulate LLM APIs.

\subsection{Problem Statement}

Current literature and benchmarks focus exclusively on characteristics such as model intelligence, accuracy, and reasoning metrics (e.g., MMLU, and GSM8k). Additionally, there is a distinct gap in evaluating metrics such as latency, throughput, reliability and cost efficiency. For this reason, developers require a method in order to understand and quantify how LLM APIs perform and behave under real-world network conditions rather than an isolated, controlled condition. We aim to change the target standard from model intelligence to system usability by introducing a tool that allows developers to benchmark API performance quantitatively in terms of latency and throughput.

\section{Literature Survey}

Current network performance measuring tools, such as Speedtest, Ping, and Traceroute, aim to evaluate and test raw network metrics and connection stability. These metrics do not include processing latency statistics related to AI workloads; An API response delay consists of a network round-trip time (RTT), as well as the variable processing time associated with token generation.

\subsection{LLM Benchmarking Current State}

As well as network performance tools, observation tools for LLMs provided by AI vendors are considered to be fragmented. While tools such as LangSmith and Helicone offer request tracing, they fail to provide a cross-vendor standardized benchmark designed for consumer evaluation. This creates a demand for a universal, client-driven benchmarking tool modeled for network bandwith testing, as well as tokenized generation streams.

\section{Approach Formulation and Execution}
\subsection{Our Approach}

To solve the aforementioned issues, we propose a web-based benchmarking tool: ``AI Speed test``. This tool will use deterministic prompts against multiple different LLM providers. Using time-based metrics and HTTPS communication, the tool calculates the Time To First Token (TTFT), Tokens per Second (TPS), and the total latency.

\subsection{Execution \& System Architecture}
In order to capture and synchronize crowdsourced network intelligence asynchronously amongst peers, we constructed a sophisticated full-stack architecture mapping the entire data lifecycle:

\begin{itemize}
    \item \textbf{Frontend Orchestrator (\texttt{useBenchmark}):} We constructed a dynamic React interface prioritizing high-granularity evaluation. Users drill down past the provider-level to target specific models (e.g., Google's \texttt{gemini-1.5-flash}). Upon execution, the orchestrator leverages browser and IP-based geolocation APIs to explicitly tag the \textit{user's} geographic origin (City, Country, Lat/Lon). Crucially, the system does \textit{not} track the specific data center serving the AI model; major providers utilize opaque global edge networks that do not expose geographic routing in their HTTP headers. Thus, our recorded latency maps the trip from the user's known origin to the provider's black-box edge node.
    
    \item \textbf{API Proxies \& Telemetry (\texttt{/api/benchmark}):} Next.js Route Handlers securely broker requests to the provider SDKs via Server-Sent Events (SSE). The system measures atomic timestamp differentials across the TCP stream, identifying exactly when the initial byte is returned (TTFT) and mathematically aggregating the subsequent stream velocity (TPS).
    
    \item \textbf{Centralized Database Aggregation (\texttt{Turso \& libSQL}):} To solve the problem of isolated local benchmarks, the application pipes telemetry directly into a distributed, serverless Turso SQLite database. This allows distributed teammates spanning different network configurations and paid API tier access to asynchronously contribute to a centralized global scoreboard. The \texttt{/api/results} endpoint then executes \texttt{GROUP BY} queries across provider, model, and region to instantly populate the application's Community Results dashboard.
\end{itemize}

\subsection{Network Layer Connections}

This project heavily emphasizes the Application layer within HTTP REST architectures, but directly relates to constraints via the Transport Layer. In any case, the Time to First Token (TTFT) is affected by the TCP handshake, as well as RTT to distributed CDNs or edge endpoints in charge of handling the LLM routing. Furthermore, the geographic location tag allows us to observe the behavior of CDN routiting among various different models, outlining key differences between LLMs.

\subsection{Future Analysis \& Visualization Strategy}
In our final presentation and expanded metrics, we will conduct quantitative analysis comparing:
\begin{enumerate}
    \item \textbf{Regional Latency Variance:} Demonstrating how RTT impacts TTFT across regions using plotted choropleth maps or bar charts correlating distance to CDN with response delay.
    \item \textbf{Hardware Processing vs Network Delay:} Using Groq (LPU hardware) against traditional GPU architectures (OpenAI/Google) to mathematically isolate the "processing" delay from the "Transport" delay.
    \item \textbf{Model Complexity Scalability:} Observing the degradation of TPS between lightweight models (\texttt{gpt-4o-mini}) and deep reasoning models (\texttt{o1-mini}) on identical network boundaries.
\end{enumerate}

\section{Conclusion}

Our proposed ``AI Speed Test`` provides the missing capability that modern speed testers and AI benchmarks lack: an objective and comparable method to benchmarking different generative API endpoints. By formulating LLM benchmarks through traditional network streaming metrics, as well as crowd-sourcing regional latency via Turso, we provide a foundation for the production of AI deployment architectures.
